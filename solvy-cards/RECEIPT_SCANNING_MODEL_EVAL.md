## 1. Receipt Region Detection

### Option A: Mock / Whole-Image Detection (current)
- **Pros:** Zero model size; OCR works immediately; fast.
- **Cons:** Fails when receipt is small/rotated/cropped in the frame.
- **Use:** Development fallback and baseline.

### Option B: Ultralytics YOLOv8n-OBB / YOLO11n-OBB (selected)
- **Pros:**
  - State-of-the-art oriented bounding boxes for rotated receipts.
  - Tiny variants (~6 MB params, ~12 MB ONNX) run on mobile.
  - Easy export to ONNX, CoreML, TFLite.
  - Fits SOLVY's Sovereign AI goal: self-hosted, no API.
- **Cons:**
  - Requires training data (~200–500 receipt images).
  - Need to handle ONNX runtime in browser and native delegates in Capacitor.
- **Training data:**
  - 200 synthetic training images + 40 validation images generated by `ml/receipt-detection/scripts/generate_synthetic_receipts.py`.
- **Results:**
  - mAP50: 0.878
  - mAP50-95: 0.385
  - ONNX export: 11.9 MB
- **Export targets:**
  - Web: `yolov8n-obb.onnx` with `onnxruntime-web` ✅ implemented
  - iOS: `yolov8n-obb.mlpackage` (CoreML) — future
  - Android: `yolov8n-obb.tflite` (NNAPI / GPU delegate) — future

### Option C: Classical Document Edge Detection (OpenCV.js)
- **Pros:** No training required; works on high-contrast receipts.
- **Cons:** Brittle with shadows, folds, cluttered backgrounds; large OpenCV.js bundle.
- **Verdict:** Skip — YOLO is more robust and aligned with Sovereign AI stack.

### Recommendation
Use **Ultralytics YOLOv8n-OBB** trained on receipt images. Start with mock detection to validate the full OCR/extraction pipeline, then swap in the ONNX model.

## 2. OCR Engine

### Option A: Tesseract.js (current)
- **Pros:**
  - Runs entirely in browser/Capacitor.
  - Mature, easy to integrate.
  - ~10 MB traineddata file (eng).
- **Cons:**
  - Accuracy drops on noisy/curved/faded receipts.
  - Slower than modern OCR models.
- **Status:** Integrated and building successfully.

### Option B: PaddleOCR (ONNX)
- **Pros:**
  - Higher accuracy on receipts and scene text.
  - Lightweight PP-OCRv3/v4 models.
  - ONNX Runtime deployment.
- **Cons:**
  - More complex integration (detection + recognition models).
  - Need to manage model files.

### Option C: EasyOCR
- **Pros:** Good accuracy.
- **Cons:** PyTorch-based; not browser-friendly. Requires server or native bridge.
- **Verdict:** Skip for on-device POC.

### Recommendation
Keep **Tesseract.js** for the POC to prove the end-to-end flow. Evaluate **PaddleOCR-on-ONNX** as a Phase 2 accuracy upgrade if receipt OCR quality is insufficient.

## 3. Bundle & Performance Budget

| Component | Size | Notes |
|-----------|------|-------|
| Tesseract.js core + eng traineddata | ~10 MB | Lazy-load traineddata from public path |
| YOLOv8n-OBB ONNX | ~12 MB | Could quantize to INT8 to reduce size |
| ONNX Runtime Web wasm | ~13–26 MB | Copied to `public/onnx-wasm/`; loaded on demand |
| Total vision bundle | ~35–48 MB | Large; acceptable for dedicated feature; lazy-load |

**Target:** Detection + OCR in under 3 seconds on mid-range smartphone.

**Current status:** Build succeeds; runtime testing required on actual device/browser.

## 4. Privacy & Sovereignty Notes

- All processing happens on the member's device.
- Tesseract.js and YOLO models run locally; no network calls for OCR/detection.
- Raw receipt images are not persisted by `receipt-processor.js`.
- Only structured metadata is stored in localStorage (POC) / IndexedDB (production).
- Future: move from localStorage to `solvy-platform/js/services/db.js` for cross-app integration.
